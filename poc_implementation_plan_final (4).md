# Redrob Hackathon — POC Implementation Plan

> **Perspective:** CTO + Lead ML Engineer
> **Core philosophy:** Build a true AI-driven system that reasons semantically about candidate fit — not a keyword matcher with a Python dictionary disguised as intelligence.

> **LLM provider:** Both LLM calls in Phase A — JD analysis (Step 4) and candidate evaluation (Step 9) — use **Groq** (`langchain-groq` + `GROQ_API_KEY`). One key, one SDK, one rate-limit budget across the whole pipeline.

> **Phase A execution model:** Every file in `pre_computation/` exposes a single `run()` function built from small, single-purpose helper functions. `pre_computation/pipeline.py` imports those functions directly and calls them in sequence — it does **not** shell out via `subprocess`. This means real Python tracebacks on failure, the ability to drop a `breakpoint()` inside any step, and the ability to run or debug one step in isolation from a REPL without touching the rest of the pipeline. See Step 11 for the full rationale.

## Scoring Architecture & Design Rationale

> Read this before touching any code. It tells you what each Phase A file is actually producing, how that output flows into the final ranking, and why the pipeline is split the way it is.

### The three scores and their weights

```
final_score = 0.65 × llm_score  +  0.20 × semantic_score  +  0.15 × behavioral_score
```

These weights live in `artifacts/rank_config.json` and can be adjusted without re-running Phase A. The defaults above reflect the relative trust the system places in each signal.

| Score | Produced by | Default weight | Why that weight |
|---|---|---|---|
| `llm_score` | `evaluate_candidates.py` (Step 9) | **65%** | The dominant signal — the LLM actually reads the JD's requirements and each candidate's career text and makes a calibrated judgment |
| `semantic_score` | `build_shortlist.py` (Step 8) | **20%** | Useful signal but blunt — embedding similarity cannot tell whether someone shipped the work or just described it |
| `behavioral_score` | `util/behavioral.py` (Step 7) | **15%** | Availability/reachability — a great candidate who's been inactive for 6 months and ignores recruiters is not actually hirable |

### What each file does, why it exists, and where its output goes

**`build_shortlist.py` — Retrieval, not ranking**

Runs cosine similarity between the JD embedding and all 100K candidate embeddings in one vectorized numpy operation. This IS the genuine semantic search across the whole pool. But embedding similarity is a blunt instrument: two candidates can score nearly identically if they both mention similar concepts ("Python," "production," "LLM"), even if one actually shipped the work and the other only describes it superficially. It cannot make that distinction. Its job is narrowing 100K → 300 candidates worth spending LLM tokens on — not deciding final order. The `semantic_score` it computes is **not discarded**; it survives as a 20%-weight input to `assemble_ranking.py`.

**`evaluate_candidates.py` — Judgment, not just similarity**

This is where genuine fit assessment happens. The LLM reads each shortlisted candidate's real career history text against the JD's specific `hard_requirements` and `disqualifier_patterns`, then produces a calibrated `llm_score` (0–1) plus a `reasoning` string grounded in that candidate's actual profile facts. This is what catches what cosine similarity cannot — for example, distinguishing someone who built production ML systems from someone who only called a hosted LLM API and happened to use similar vocabulary. Its score carries **65% weight** — the dominant input to the final ranking. The `reasoning` string it produces is carried forward unchanged through `assemble_ranking.py` and `rank.py` into the final CSV. Nothing downstream edits or regenerates it.

**`util/behavioral.py` — Availability, not fit**

Scores each candidate on how reachable they are right now, independently of what the JD says. Last active date, open-to-work flag, recruiter response rate, interview completion rate, notice period. A technically perfect candidate who hasn't logged in for 7 months and ignores recruiters is practically unhirable regardless of their skills. Its score carries **15% weight**. This function is called once per candidate during `build_shortlist.py` (for all 100K — it's fast) and the results are saved to `behavioral_scores.json` for `assemble_ranking.py` to use.

**`assemble_ranking.py` — Combination, not computation**

Loads the three scores from their respective artifact files, blends them using the default weights, sorts the result, and writes `final_ranking.json`. It also carries the LLM's `reasoning` string through unchanged into the output — it does not regenerate or modify it. This is the only file that knows all three scores exist; each upstream step is ignorant of the others. `rank_config.json` is also written here as a record of which weights produced this ranking.

**`rank.py` (Phase B) — Replay, not generation**

Loads `final_ranking.json` (written hours earlier by Phase A), recomputes the weighted blend in case `rank_config.json` was manually retuned since Phase A ran, re-sorts by `(-score, candidate_id)` for tie-breaking, and writes the CSV. No network access, no LLM calls, no embedding model. Completes in seconds because it is replaying frozen Phase A output, not generating anything new. **One consequence worth knowing:** if you retune weights and re-run `rank.py`, `score` and rank position can shift, but the `reasoning` text attached to each candidate will not. It is frozen from Step 9 and is independent of whichever weighting formula wins later.

### Is this design what the judges expect?

Yes — checked directly against the requirements:
- Compute constraints are explicitly scoped: *"Pre-computation may exceed the 5-minute window — only the ranking step that produces the CSV must finish within 5 minutes."*
- Stage 3 reproduces the **ranking step** in a sandboxed Docker, not the entire pipeline.
- Section 11.3 names this pattern explicitly: *"Pre-computed artifacts (embeddings, indexes, model weights), or a script to generate them."* Committing `final_ranking.json` and providing `pipeline.py` as the generator is exactly that.

### Two known gaps (not yet fixed in code)

1. **Hard requirements vs. disqualifiers are scored on one rubric.** The JD says missing any hard requirement (production embeddings, vector DB, Python, ranking evaluation) should disqualify — but the current `EVAL_PROMPT` 0.85–0.00 gradient treats hard requirements and disqualifier patterns with the same severity, so a candidate missing one hard requirement but strong elsewhere could land in the "minor gaps" band instead of being floored. Splitting `EVAL_PROMPT` into separate rubric sections for each category would better match the specification.
2. **Honeypot avoidance is assumed, not verified.** The specification expects semantic reasoning to naturally avoid the ~80 honeypot profiles (>10% in the top 100 = Stage 3 disqualification). The pipeline is plausibly capable of this — full career history and dates reach the LLM — but it has not been tested against honeypot-shaped profiles. Validate this once real data is available by checking whether profiles with impossible date overlaps or expert-level skills with 0 duration months appear in the top 100.

### STEP 4 — Phase A: JD Analysis via Groq LLM (`pre_computation/analyze_jd.py`)

**Objective:** Dynamically extract a semantic profile from the JD using an LLM. This replaces the manual `jd_config.py` approach entirely. The output is a structured JSON that captures *what the role means* — not a list of keywords.

**Why this is the correct approach:** The JD explicitly warns against keyword matching. A hardcoded keyword dictionary is brittle to synonym variation, domain jargon, and alternative phrasings. An LLM reading the full JD understands that "built a semantic retrieval pipeline" and "developed RAG infrastructure" describe the same capability, even with zero keyword overlap.

**This step also introduces `util/llm_client.py`** — a small shared module that builds the Groq client and parses JSON out of LLM responses (stripping markdown code fences if present). Step 9 reuses this same module rather than duplicating the API-key validation and fence-stripping logic a second time.

```python
# util/llm_client.py
"""
Shared Groq LLM client construction and response parsing.
Used by both pre_computation/analyze_jd.py and pre_computation/evaluate_candidates.py
so the API key handling and JSON-extraction logic live in exactly one place.
"""
import json
import os

from dotenv import load_dotenv
from langchain_groq import ChatGroq

DEFAULT_GROQ_MODEL = 'llama-3.3-70b-versatile'


def build_groq_llm(model: str = DEFAULT_GROQ_MODEL, max_tokens: int = 2000) -> ChatGroq:
    """
    Loads GROQ_API_KEY from .env / the environment and constructs a ChatGroq client.
    Raises a clear, immediate error if the key is missing, rather than failing
    later with an opaque auth error on the first API call.
    """
    load_dotenv()
    groq_api_key = os.getenv("GROQ_API_KEY")
    if not groq_api_key:
        raise ValueError(
            "GROQ_API_KEY is not set. Get a free key at https://console.groq.com/keys "
            "and add it to your .env file."
        )
    return ChatGroq(groq_api_key=groq_api_key, model_name=model, max_tokens=max_tokens)


def parse_json_response(raw: str):
    """
    Strips markdown code fences (```json ... ```) that LLMs sometimes wrap
    JSON output in, then parses. Works for both a single JSON object (step1's
    jd_profile) and a JSON array (step4's batch evaluation) — json.loads
    handles either shape once the fences are gone.
    """
    raw = raw.strip()
    if raw.startswith("```"):
        raw = raw.split("```", 1)[1]
        if raw.startswith("json"):
            raw = raw[4:]
        raw = raw.rsplit("```", 1)[0]
    return json.loads(raw.strip())
```

**Implementation Actions:**

```python
# pre_computation/analyze_jd.py
"""
Phase A - Step 1: Extract semantic JD profile via Groq LLM.

Public entrypoint: run(jd_path, artifacts_dir) -> dict (the jd_profile)

Outputs:
  artifacts/jd_profile.json   — structured semantic context for LLM evaluator
  artifacts/jd_embedding.npy  — 384-dim vector for embedding similarity search
"""
import json
import os
from pathlib import Path

import numpy as np
from docx import Document
from sentence_transformers import SentenceTransformer

from util.llm_client import build_groq_llm, parse_json_response

ARTIFACTS_DIR = 'artifacts'
GROQ_MODEL = 'llama-3.3-70b-versatile'   # confirm current model name at console.groq.com/docs/models

JD_EXTRACTION_PROMPT = """You are analyzing a job description to power an AI candidate ranking system.

Read this job description carefully — including any sections addressed to hackathon participants or systems being built to evaluate it. Pay special attention to what the role explicitly says it does NOT want, and what it says about how candidates should be evaluated.

Extract the following and return as a single valid JSON object. No other text.

{{
  "embedding_text": "<200-250 word dense prose written in the PRESENT TENSE describing what an ideal candidate HAS DONE and CAN DO in this role. Write it from the candidate's perspective — as if it is the ideal candidate's career summary. Optimize for semantic vector comparison against real candidate profiles. Focus on actual work done, systems built, and impact delivered — not job title words or keyword lists.>",

  "role_summary": "<2-3 sentence plain-English description of what this person does day to day in this role — useful for explaining the role to an LLM evaluator who has never seen the JD>",

  "hard_requirements": [
    "<Specific technical capability the candidate must have DEMONSTRATED — describe the actual work, not just a technology name. E.g. 'Shipped a production embedding-based retrieval system that served real user queries at scale' rather than 'knows sentence-transformers'>",
    ...list all hard requirements this way...
  ],

  "disqualifier_patterns": [
    "<Career pattern that makes a candidate unsuitable — describe the pattern, not the technology. E.g. 'Entire career at IT outsourcing/consulting firms with no product company experience' or 'Current AI experience is limited to calling hosted LLM APIs via LangChain without any pre-LLM production ML depth'>",
    ...list all disqualifiers this way...
  ],

  "preferred_location": "<City/region preferences from the JD, as a plain string>",

  "experience_years": {{"min": <integer>, "max": <integer>}},

  "notice_preference": "<What the JD says about notice period, as a plain string>",

  "evaluation_guidance": "<2-3 sentences of guidance for the LLM that will score individual candidates — summarizing the spirit of what this JD is really looking for, including any warnings about traps or anti-patterns to avoid when scoring>"
}}

Job Description:
{jd_text}"""


def read_jd(path: str) -> str:
    """Read JD from .docx or .txt/.md file. No side effects beyond the file read."""
    p = Path(path)
    if p.suffix.lower() == '.docx':
        doc = Document(path)
        return '\n'.join(para.text for para in doc.paragraphs if para.text.strip())
    return p.read_text(encoding='utf-8')


def extract_jd_profile(jd_text: str, llm) -> dict:
    """One LLM call -> parsed dict. No file I/O — easy to call directly in a test or REPL."""
    response = llm.invoke(JD_EXTRACTION_PROMPT.format(jd_text=jd_text))
    return parse_json_response(response.content)


def embed_jd_profile(jd_profile: dict, embedder: SentenceTransformer) -> np.ndarray:
    """Embeds the profile's embedding_text field. No file I/O."""
    return embedder.encode(
        [jd_profile['embedding_text']],

### STEP 5 — Candidate Text Construction (`util/candidate_text.py`)

**Objective:** Define how a candidate's profile is converted into text for embedding. This is the most consequential design decision in the embedding pipeline. The text fed to the embedding model determines what the semantic vector represents.

**Design principle:** Career history descriptions are the primary signal — they describe what a person actually did, in their own words or their employer's words. They are harder to game than a skills list. The embedding model compares the *meaning* of these descriptions against the *meaning* of the JD. No keywords are checked anywhere in this function.

```python
# util/candidate_text.py
"""
Converts a candidate profile into a dense text representation for embedding.

Design decisions:
- Career descriptions are the LEAD signal — placed first, weighted most by the embedding
- Summary and headline provide context
- Skills are included only if they carry evidence (endorsements > 0 OR duration > 0)
  This is a data-quality filter, not a keyword filter — it removes content with zero evidence
- No keyword matching occurs here. The embedding model determines semantic relevance.
- Recent roles are placed before older ones (most candidate records already order this way)
"""

def build_candidate_text(candidate: dict) -> str:
    parts = []
    profile = candidate['profile']

    # 1. Headline and summary — context for the embedding
    headline = profile.get('headline', '').strip()
    summary = profile.get('summary', '').strip()
    if headline:
        parts.append(headline)
    if summary:
        parts.append(summary)

    # 2. Career descriptions — the primary semantic signal
    # career_history is already ordered most-recent-first in the schema
    for job in candidate.get('career_history', []):
        title = job.get('title', '').strip()
        company = job.get('company', '').strip()
        description = job.get('description', '').strip()
        if description:
            # Include title and company for context, then the full description
            parts.append(f"Role: {title} at {company}. {description}")

    # 3. Skills with at least one piece of evidence
    # endorsements > 0 OR duration_months > 0 means SOMETHING supports this skill
    # endorsements == 0 AND duration_months == 0 means someone typed a keyword and nothing else
    evidenced_skills = [
        s['name'] for s in candidate.get('skills', [])
        if s.get('endorsements', 0) > 0 or s.get('duration_months', 0) > 0
    ]
    if evidenced_skills:
        parts.append(f"Skills with evidence: {', '.join(evidenced_skills)}")

    return ' '.join(part for part in parts if part)
```

### STEP 6 — Phase A: Batch Embedding of All Candidates (`pre_computation/embed_candidates.py`)

**Objective:** Encode all 100K candidates into 384-dimensional vectors using the same model that embedded the JD. This enables cosine similarity comparison at scale.

```python
# pre_computation/embed_candidates.py
"""
Phase A - Step 2: Embed all 100K candidates using all-MiniLM-L6-v2.

Public entrypoint: run(candidates_file, artifacts_dir, batch_size) -> None

Outputs:
  artifacts/candidate_embeddings.npy  — float32 (100K, 384), ~153 MB
  artifacts/candidate_ids.txt          — ordered list of CAND_XXXXXXX
"""
import json
import os

import numpy as np
from tqdm import tqdm
from sentence_transformers import SentenceTransformer

from util.candidate_text import build_candidate_text

CANDIDATES_FILE = 'data/candidates.jsonl'
ARTIFACTS_DIR = 'artifacts'
BATCH_SIZE = 128  # reduce to 32 if you run out of memory


def load_candidate_texts(candidates_file: str) -> tuple:
    """
    Reads the full candidate file once. Returns (candidate_ids, texts), in matching order.
    Isolated from embedding so you can sanity-check text construction without
    paying the embedding cost: load_candidate_texts(...) and print texts[:5].
    """
    candidate_ids, texts = [], []
    with open(candidates_file, 'r', encoding='utf-8') as f:
        for line in tqdm(f, total=100_000, desc='Reading candidates'):
            c = json.loads(line)
            candidate_ids.append(c['candidate_id'])
            texts.append(build_candidate_text(c))
    return candidate_ids, texts


def embed_texts(texts: list, model: SentenceTransformer, batch_size: int = BATCH_SIZE) -> np.ndarray:
    """Pure embedding call — no file I/O. Easy to unit test on a handful of strings."""
    return model.encode(
        texts,
        batch_size=batch_size,
        show_progress_bar=True,
        convert_to_numpy=True,
        normalize_embeddings=True,   # L2 normalise → dot product = cosine similarity
    )


def run(candidates_file: str = CANDIDATES_FILE, artifacts_dir: str = ARTIFACTS_DIR,
        batch_size: int = BATCH_SIZE) -> None:
    os.makedirs(artifacts_dir, exist_ok=True)

    print("Loading sentence-transformer model (all-MiniLM-L6-v2)...")
    model = SentenceTransformer('all-MiniLM-L6-v2')

    print("Reading all candidates and building text representations...")
    candidate_ids, texts = load_candidate_texts(candidates_file)

    print(f"\nEncoding {len(texts)} candidates in batches of {batch_size}...")
    embeddings = embed_texts(texts, model, batch_size)

    print(f"\nSaving artifacts...")
    np.save(f'{artifacts_dir}/candidate_embeddings.npy', embeddings)
    with open(f'{artifacts_dir}/candidate_ids.txt', 'w') as f:
        f.write('\n'.join(candidate_ids))

    print(f"✓ candidate_embeddings.npy: {embeddings.shape}, {embeddings.nbytes / 1e6:.0f} MB")
    print(f"✓ candidate_ids.txt: {len(candidate_ids)} IDs")
    print(f"\nEmbedding stats: min={embeddings.min():.3f}, "
          f"mean={embeddings.mean():.3f}, max={embeddings.max():.3f}")


if __name__ == '__main__':
    run()

### STEP 7 — Phase A: Behavioral Scoring (`util/behavioral.py`)

**Objective:** Compute a data-driven behavioral score for each candidate using the 23 platform signals. This is not a JD-specific rule set — it captures universal hiring signals that apply to any role: is this candidate reachable, available, and actively engaging?

> **Score produced:** `behavioral_score` — a multiplier-based float \[0, 1\] reflecting how hirable the candidate is *right now*, independent of any JD content. Inputs: activity recency, open-to-work flag, recruiter response rate, interview completion rate, notice period, profile completeness.  
> **Final ranking weight:** **15%** (`behavioral_weight` in `rank_config.json`)  
> **Flows to:** `behavioral_scores.json` → `assemble_ranking.py` → blended into `final_score`  
> **Called by:** `build_shortlist.py` (once per candidate across all 100K, fast) and re-loaded by `assemble_ranking.py`

**Why this is legitimate and not "hardcoded rules":** Behavioral scoring uses the actual numeric values from `redrob_signals` and applies thresholds that reflect practical hiring realities — not JD content. The same function produces meaningful scores for any role. It never checks for keywords, job titles, or company names.

```python
# util/behavioral.py
"""
Data-driven behavioral scorer using redrob_signals platform data.
Returns a float [0, 1] representing how "hirable" a candidate is right now.
This score is universal — it does not depend on the JD content.
"""
from datetime import date

def compute_behavioral_score(candidate: dict) -> float:
    s = candidate.get('redrob_signals', {})
    today = date.today()
    score = 1.0

    # ── Activity recency ─────────────────────────────────────────────────────
    # A candidate who hasn't logged in for 6+ months is practically unreachable
    try:
        last_active = date.fromisoformat(s.get('last_active_date', '2020-01-01'))
        days_inactive = (today - last_active).days
    except (ValueError, TypeError):
        days_inactive = 999

    if days_inactive > 180:
        score *= 0.30
    elif days_inactive > 90:
        score *= 0.65
    elif days_inactive > 45:
        score *= 0.85

    # ── Active job search ────────────────────────────────────────────────────
    if not s.get('open_to_work_flag', False):
        score *= 0.75

    # ── Recruiter reachability ───────────────────────────────────────────────
    rr = s.get('recruiter_response_rate', 0.5)
    if rr < 0.20:
        score *= 0.50   # historically ignores most outreach
    elif rr < 0.40:
        score *= 0.80

    # ── Interview reliability ────────────────────────────────────────────────
    icr = s.get('interview_completion_rate', 0.5)
    if icr < 0.50:
        score *= 0.80   # historically ghosts scheduled interviews

    # ── Notice period ────────────────────────────────────────────────────────
    # JD: sub-30 days preferred; 30+ still in scope but bar is higher (§2.4).
    # This used to be sent to evaluate_candidates.py as part of behavioral_signals,
    # costing LLM tokens for a signal the EVAL_PROMPT rubric never referenced.
    # It belongs here instead: deterministic, free, and already weighted into
    # the final score via behavioral_weight in assemble_ranking.run().
    notice_days = s.get('notice_period_days', 60)
    if notice_days <= 30:
        pass  # no penalty — meets JD preference
    elif notice_days <= 45:
        score *= 0.90
    elif notice_days <= 60:
        score *= 0.80
    else:
        score *= 0.65

    # ── Profile completeness ─────────────────────────────────────────────────
    # Incomplete profiles provide less data for evaluation — lower confidence
    completeness = s.get('profile_completeness_score', 50) / 100.0
    score *= (0.70 + 0.30 * completeness)

    return min(max(score, 0.0), 1.0)
```

### STEP 8 — Phase A: First-Pass Shortlisting (`pre_computation/build_shortlist.py`)

**Objective:** Use embedding similarity and behavioral scores to narrow 100K candidates down to a shortlist of 300 for LLM evaluation. This step is fast and purely numeric — no LLM API calls, no keyword checks.

> **Score produced:** `semantic_score` — cosine similarity between the candidate's embedding and the JD embedding, normalised to \[0, 1\]  
> **Final ranking weight:** **20%** (`semantic_weight` in `rank_config.json`)  
> **Flows to:** carried inside `shortlist.jsonl` as the `_semantic_score` field → `assemble_ranking.py` → blended into `final_score`  
> **This step also produces:** `behavioral_scores.json` for all 100K candidates — loaded by `assemble_ranking.py` (not computed again)

**Why 300 for the POC — and when to scale up:** The LLM evaluation step (Step 9) evaluates these candidates in batches of `BATCH_SIZE` (8 by default — see Step 9). At 300 candidates that's ~38 API calls — feasible within Groq's free-tier token limits in a single session without hitting rate exhaustion. Once the full pipeline is verified end-to-end and working, increase `SHORTLIST_SIZE` to 800–1,000 for the actual submission run. You can do that across multiple sessions since `pipeline.py` skips any step whose artifact files already exist — so re-running after increasing the shortlist only re-runs Steps 3 onward, not the 15-minute embedding step.

> **Rate limit tip:** Step 9 already defaults to the lower-cost configuration (`llama-3.1-8b-instant`, trimmed candidate payload, `BATCH_SIZE=8`) — that's not a lever you have left to pull. If you still hit Groq token exhaustion at 300 candidates, the remaining options are: (1) increase `SLEEP_BETWEEN_BATCHES` in `evaluate_candidates.py` from 2s to 5s if it's a per-minute cap; (2) shrink `FULL_DETAIL_ROLES` or `MAX_TEXT_CHARS` in `evaluate_candidates.py` to trim tokens further if it's a per-day cap; or (3) just resume the next day — `evaluate_candidates.run()` checkpoints after every batch and skips candidates already scored, so re-running `python -m pre_computation.evaluate_candidates` picks up exactly where it left off instead of re-spending tokens on completed work.

```python
# pre_computation/build_shortlist.py
"""
Phase A - Step 3: First-pass shortlisting via embedding similarity + behavioral score.

Public entrypoint: run(...) -> None

Outputs:
  artifacts/shortlist.jsonl           — top N full candidate records + pre-scores
  artifacts/behavioral_scores.json   — {cid: behavioral_score} for all 100K
"""
import json
import os

import numpy as np
from tqdm import tqdm

from util.behavioral import compute_behavioral_score

CANDIDATES_FILE = 'data/candidates.jsonl'
ARTIFACTS_DIR = 'artifacts'
SHORTLIST_SIZE = 300  # start small for POC — scale to 800-1000 for final submission

# Weights for first-pass shortlisting ONLY (not final ranking — see assemble_ranking.py)
SHORTLIST_EMBEDDING_WEIGHT = 0.75
SHORTLIST_BEHAVIORAL_WEIGHT = 0.25


def compute_semantic_scores(embeddings: np.ndarray, jd_embedding: np.ndarray) -> np.ndarray:
    """
    Cosine similarity for all candidates at once (both inputs are L2-normalised,
    so dot product = cosine similarity). Mapped from [-1,1] to [0,1].
    Pure function — easy to test on a tiny fake (N, 384) array.
    """
    raw = (embeddings @ jd_embedding).astype(np.float64)
    return (raw + 1.0) / 2.0


def compute_all_behavioral_scores(candidates_file: str) -> dict:
    """One pass through the full candidate file. Returns {candidate_id: score}."""
    scores = {}
    with open(candidates_file, 'r', encoding='utf-8') as f:
        for line in tqdm(f, total=100_000, desc='Behavioral scoring'):
            c = json.loads(line)
            scores[c['candidate_id']] = compute_behavioral_score(c)
    return scores


def select_top_ids(candidate_ids: list, combined_scores: np.ndarray, size: int) -> set:
    """Returns the set of candidate_ids with the highest combined_scores."""
    top_indices = np.argsort(combined_scores)[::-1][:size]
    return {candidate_ids[i] for i in top_indices}


def collect_shortlist_records(candidates_file: str, top_ids: set, semantic_by_id: dict,
                               behavioral_by_id: dict, combined_by_id: dict) -> list:
    """Second pass: pulls full records for only the shortlisted IDs, attaches pre-scores."""
    shortlist = []
    with open(candidates_file, 'r', encoding='utf-8') as f:
        for line in tqdm(f, total=100_000, desc='Collecting shortlist'):
            c = json.loads(line)
            cid = c['candidate_id']
            if cid in top_ids:
                c['_semantic_score'] = semantic_by_id[cid]
                c['_behavioral_score'] = behavioral_by_id[cid]
                c['_combined_score'] = combined_by_id[cid]
                shortlist.append(c)
    shortlist.sort(key=lambda x: x['_combined_score'], reverse=True)
    return shortlist


def run(candidates_file: str = CANDIDATES_FILE, artifacts_dir: str = ARTIFACTS_DIR,
        shortlist_size: int = SHORTLIST_SIZE) -> None:
    print("Loading embeddings and JD vector...")
    embeddings = np.load(f'{artifacts_dir}/candidate_embeddings.npy')
    with open(f'{artifacts_dir}/candidate_ids.txt') as f:
        candidate_ids = [line.strip() for line in f if line.strip()]
    jd_embedding = np.load(f'{artifacts_dir}/jd_embedding.npy')

    print("Computing semantic similarity for all 100K candidates...")
    semantic_scores = compute_semantic_scores(embeddings, jd_embedding)
    print(f"Semantic score range: [{semantic_scores.min():.4f}, {semantic_scores.max():.4f}]")

    print("Computing behavioral scores (one pass through candidates)...")
    behavioral_scores = compute_all_behavioral_scores(candidates_file)

    semantic_by_id = {cid: float(semantic_scores[i]) for i, cid in enumerate(candidate_ids)}
    combined_by_id = {
        cid: SHORTLIST_EMBEDDING_WEIGHT * semantic_by_id[cid]
             + SHORTLIST_BEHAVIORAL_WEIGHT * behavioral_scores.get(cid, 0.5)
        for cid in candidate_ids
    }
    combined_arr = np.array([combined_by_id[cid] for cid in candidate_ids])

    top_ids = select_top_ids(candidate_ids, combined_arr, shortlist_size)
    print(f"\nTop {shortlist_size} shortlist combined score range: "
          f"[{min(combined_by_id[c] for c in top_ids):.4f}, "
          f"{max(combined_by_id[c] for c in top_ids):.4f}]")

    print(f"Collecting full records for {shortlist_size} candidates (second pass)...")
    shortlist = collect_shortlist_records(
        candidates_file, top_ids, semantic_by_id, behavioral_scores, combined_by_id
    )

    with open(f'{artifacts_dir}/shortlist.jsonl', 'w') as f:
        for c in shortlist:
            f.write(json.dumps(c) + '\n')

    with open(f'{artifacts_dir}/behavioral_scores.json', 'w') as f:
        json.dump(behavioral_scores, f)

    print(f"\n✓ shortlist.jsonl: {len(shortlist)} candidates")
    print(f"✓ behavioral_scores.json: {len(behavioral_scores)} entries")


if __name__ == '__main__':
    run()

### STEP 9 — Phase A: LLM Batch Evaluation of Shortlist (`pre_computation/evaluate_candidates.py`)

**Objective:** Use the LLM to deeply evaluate each of the shortlisted candidates against the dynamically extracted JD profile. This is the core intelligence of the system — it produces both a precise relevance score and a unique, fact-specific reasoning string for every candidate.

> **Score produced:** `llm_score` (float 0–1) — the LLM's judgment of how well the candidate's actual career history matches the JD's requirements. Also produces a `reasoning` string that is carried forward **unchanged** all the way into the final CSV.  
> **Final ranking weight:** **65%** (`llm_weight` in `rank_config.json`) — the dominant signal  
> **Flows to:** `llm_evaluations.json` → `assemble_ranking.py` → blended into `final_score`, `reasoning` passed through into `final_ranking.json` and the CSV  
> **Note:** The `reasoning` text is frozen at this step. Re-tuning weights in `rank_config.json` and re-running `rank.py` changes score and rank position but does NOT change the reasoning text.

**Why this produces non-templated reasoning:** The LLM reads the candidate's actual career description text and generates reasoning grounded in those specific facts. Two candidates with similar embedding scores will receive completely different reasoning because their career descriptions are different. No two outputs of this function will be identical.

**Provider note:** This script reuses `util/llm_client.py` — same `GROQ_API_KEY`, same `build_groq_llm()` helper. It deliberately uses a **different model** from `analyze_jd.py`: `llama-3.1-8b-instant` instead of `llama-3.3-70b-versatile`. Reasoning: JD analysis runs once and output quality matters most there — use the best model. Candidate evaluation runs 40–100× more often and is the volume step — use the model with the highest free-tier token budget.

```python
# pre_computation/evaluate_candidates.py
"""
Phase A - Step 4: LLM batch evaluation of shortlisted candidates.

Public entrypoint: run(...) -> dict (results, also saved to llm_evaluations.json)

Uses Groq's llama-3.1-8b-instant via util.llm_client — same GROQ_API_KEY as
analyze_jd.py, but deliberately a smaller model: this step makes 40-100x more
API calls than JD analysis, so it needs the tier with the larger free-tier
token budget, not the most capable one. Reserve llama-3.3-70b-versatile for
analyze_jd.py, which runs exactly once and is where quality matters most.

TOKEN BUDGET, NOT JUST RATE LIMIT:
Groq's free tier has both a per-minute cap (RPM/TPM) and a per-day token cap.
Slowing down requests (SLEEP_BETWEEN_BATCHES, backoff) only helps the
per-minute cap. It does NOT help a per-day cap: if that's the binding
constraint, the only fix is fewer tokens per call or resuming on a later day.
This script reduces tokens at the source (trimmed payload, no redundant fields,
amortized batch overhead) AND backs off correctly so it does not waste retries
against a limit retrying cannot fix.
"""
import re
import json
import time

from tqdm import tqdm

from util.llm_client import build_groq_llm, parse_json_response

ARTIFACTS_DIR = "artifacts"
BATCH_SIZE = 8                    # up from 5 — amortizes fixed JD-context cost across more candidates
SLEEP_BETWEEN_BATCHES = 2.0       # seconds — courtesy gap for per-minute limits
MAX_RETRIES = 5
LONG_WAIT_THRESHOLD_S = 90        # wait longer than this = likely daily cap, not per-minute
GROQ_MODEL = "llama-3.1-8b-instant"   # smaller model = much higher free-tier token budget

# career_history is the single biggest token cost in the prompt by a wide margin.
FULL_DETAIL_ROLES = 4             # most recent N roles get full description text
MAX_TEXT_CHARS = 400              # truncation length for descriptions/summary


def _truncate(text: str, max_chars: int) -> str:
    """Hard truncates at a word boundary."""
    text = (text or "").strip()
    if len(text) <= max_chars:
        return text
    return text[:max_chars].rsplit(" ", 1)[0] + "\u2026"


def build_candidate_summary(c: dict) -> dict:
    """
    Compact candidate representation for the LLM evaluator. Pure — no I/O.

    Token design:
    - Full description text only for the FULL_DETAIL_ROLES most recent roles.
      Older roles still contribute title/company/duration_months for pattern
      detection (title-chasing, tenure) — just not full prose.
    - No behavioral_signals block. Those signals are already scored for free
      and deterministically in util.behavioral.compute_behavioral_score(),
      including notice_period_days which was moved there explicitly (see Step 7).
      The EVAL_PROMPT rubric never asks the LLM to use them — sending them here
      was paying tokens for a signal already captured elsewhere.
    """
    profile = c["profile"]
    career = c.get("career_history", [])

    career_summary = []
    for i, job in enumerate(career):
        entry = {
            "title":           job.get("title", ""),
            "company":         job.get("company", ""),
            "duration_months": job.get("duration_months", 0),
            "industry":        job.get("industry", ""),
        }
        if i < FULL_DETAIL_ROLES:
            entry["description"] = _truncate(job.get("description", ""), MAX_TEXT_CHARS)
        career_summary.append(entry)

    return {
        "candidate_id":        c["candidate_id"],
        "current_title":       profile.get("current_title", ""),
        "years_of_experience": profile.get("years_of_experience", 0),
        "location":            f"{profile.get('location', '')}, {profile.get('country', '')}",
        "headline":            profile.get("headline", ""),
        "summary":             _truncate(profile.get("summary", ""), MAX_TEXT_CHARS),
        "career_history":      career_summary,
        "education": [
            {
                "degree":      edu.get("degree", ""),
                "field":       edu.get("field_of_study", ""),
                "institution": edu.get("institution", ""),
                "tier":        edu.get("tier", "unknown"),
            }
            for edu in c.get("education", [])
        ],
    }


EVAL_PROMPT = """You are a senior technical recruiter scoring candidates for a specific role.

ROLE CONTEXT:
{role_summary}

WHAT THIS ROLE REQUIRES (describe DONE work, not keywords):
{hard_requirements}

DISQUALIFYING CAREER PATTERNS (these make a candidate unsuitable):
{disqualifier_patterns}

EVALUATION GUIDANCE:
{evaluation_guidance}

ADDITIONAL CONTEXT: Location preference: {preferred_location}. Experience: {exp_min}\u2013{exp_max} years. Notice period: {notice_preference}.

Score each candidate 0.0 to 1.0 using this guide:
- 0.85\u20131.0: Career descriptions clearly demonstrate the required experience at product companies
- 0.65\u20130.84: Most requirements met, minor gaps or concerns
- 0.40\u20130.64: Relevant experience exists but significant gaps present
- 0.20\u20130.39: Tangential or adjacent experience only
- 0.00\u20130.19: Wrong domain or a disqualifying career pattern is present

For reasoning: Write 1\u20132 sentences citing SPECIFIC facts from THIS candidate's career descriptions. Do not write generic praise. Do not use identical reasoning for different candidates. Only cite facts that appear in the candidate data provided.

Return ONLY a JSON array. No other text:
[
  {{
    "candidate_id": "CAND_XXXXXXX",
    "score": <float 0.0 to 1.0>,
    "reasoning": "<1-2 sentences with specific facts>"
  }},
  ...
]

Candidates to evaluate:
{candidates_json}"""


def build_eval_prompt(batch: list, jd_profile: dict) -> str:
    """Formats EVAL_PROMPT for one batch. Pure — no I/O, no API call."""
    summaries = [build_candidate_summary(c) for c in batch]
    exp = jd_profile.get("experience_years", {"min": 5, "max": 9})
    return EVAL_PROMPT.format(
        role_summary          = jd_profile.get("role_summary", ""),
        hard_requirements     = "\n".join(f"- {r}" for r in jd_profile.get("hard_requirements", [])),
        disqualifier_patterns = "\n".join(f"- {d}" for d in jd_profile.get("disqualifier_patterns", [])),
        evaluation_guidance   = jd_profile.get("evaluation_guidance", ""),
        preferred_location    = jd_profile.get("preferred_location", "India"),
        exp_min               = exp.get("min", 5),
        exp_max               = exp.get("max", 9),
        notice_preference     = jd_profile.get("notice_preference", "under 30 days preferred"),
        candidates_json       = json.dumps(summaries, indent=2),
    )


def estimate_max_tokens(batch_size: int) -> int:
    """Scales max_tokens with actual batch size. ~120 tokens per output entry + buffer."""
    return 120 * batch_size + 250


_RETRY_AFTER_RE = re.compile(r'(?:try again in|retry after)\s*([\d.]+)\s*s', re.IGNORECASE)


def _suggested_wait_seconds(error_message: str):
    """Pulls suggested wait time out of a Groq rate-limit error message."""
    match = _RETRY_AFTER_RE.search(error_message)
    return float(match.group(1)) if match else None


def evaluate_batch(batch: list, jd_profile: dict, llm, max_retries: int = MAX_RETRIES) -> list:
    """
    One LLM call for a batch, with retry/backoff on per-minute rate-limit errors.
    Distinguishes per-minute limits (fixable by waiting) from daily caps (not fixable —
    surfaces them immediately so you can resume later rather than burning retries).
    """
    prompt = build_eval_prompt(batch, jd_profile)
    last_err = None
    for attempt in range(max_retries):
        try:
            response = llm.invoke(prompt)
            return parse_json_response(response.content)
        except Exception as e:
            last_err = e
            msg = str(e)
            if "rate" in msg.lower() or "429" in msg:
                wait = _suggested_wait_seconds(msg)
                if wait is not None and wait > LONG_WAIT_THRESHOLD_S:
                    raise RuntimeError(
                        f"Groq suggests waiting {wait:.0f}s — likely a daily token cap, "
                        f"not a per-minute one. Resume later with: "
                        f"python -m pre_computation.pipeline --from 4"
                    ) from e
                backoff = wait if wait is not None else 2 ** attempt
                print(f"  Rate limited, waiting {backoff:.0f}s (attempt {attempt + 1}/{max_retries})...")
                time.sleep(backoff)
                continue
            raise
    raise last_err


def evaluate_one_with_fallback(c: dict, jd_profile: dict, llm) -> dict:
    """Single-candidate retry used when a batch fails — isolates the broken candidate."""
    try:
        time.sleep(1)
        result = evaluate_batch([c], jd_profile, llm)[0]
        return {"score": float(result["score"]), "reasoning": result["reasoning"]}
    except Exception as e:
        print(f"  Individual eval failed for {c['candidate_id']}: {e}")
        return {
            "score":     c.get("_semantic_score", 0.0),
            "reasoning": "Automated scoring only — LLM evaluation failed for this candidate.",
        }


def _save_results(results: dict, artifacts_dir: str) -> None:
    """Writes the current results dict to disk. Called after every batch, not
    just at the end — this is the actual checkpoint. A crash, Ctrl-C, or a
    daily-token-cap RuntimeError after this point loses at most one batch's
    worth of work, not the whole run."""
    with open(f"{artifacts_dir}/llm_evaluations.json", "w") as f:
        json.dump(results, f, indent=2)


def run(artifacts_dir: str = ARTIFACTS_DIR, batch_size: int = BATCH_SIZE) -> dict:
    llm = build_groq_llm(model=GROQ_MODEL, max_tokens=estimate_max_tokens(batch_size))
    print(f"[INFO] Groq LLM initialized: {GROQ_MODEL} (max_tokens={estimate_max_tokens(batch_size)})")

    with open(f"{artifacts_dir}/jd_profile.json") as f:
        jd_profile = json.load(f)

    candidates = []
    with open(f"{artifacts_dir}/shortlist.jsonl") as f:
        for line in f:
            if line.strip():
                candidates.append(json.loads(line))

    # ── Resume support: load any partial results from a prior interrupted run ──
    # This is what makes "resume later" (the advice given when a daily-cap
    # RuntimeError fires in evaluate_batch) actually work, instead of just
    # being a suggestion that re-spends tokens on candidates already scored.
    results: dict = {}
    try:
        with open(f"{artifacts_dir}/llm_evaluations.json") as f:
            results = json.load(f)
        if results:
            print(f"[INFO] Resuming — {len(results)} candidates already evaluated in a prior run.")
    except (FileNotFoundError, json.JSONDecodeError):
        pass

    remaining = [c for c in candidates if c["candidate_id"] not in results]
    skipped = len(candidates) - len(remaining)
    if skipped:
        print(f"[INFO] Skipping {skipped} already-evaluated candidates.")

    print(f"Evaluating {len(remaining)} candidates in batches of {batch_size}...")
    print(f"Estimated API calls: {len(remaining) // batch_size + 1}")

    failed = []
    batches = [remaining[i:i + batch_size] for i in range(0, len(remaining), batch_size)]

    for batch in tqdm(batches, desc="LLM evaluation"):
        try:
            batch_results = evaluate_batch(batch, jd_profile, llm)
            for r in batch_results:
                results[r["candidate_id"]] = {"score": float(r["score"]), "reasoning": r["reasoning"]}
        except Exception as e:
            print(f"\nBatch failed ({e}). Retrying individually...")
            for c in batch:
                outcome = evaluate_one_with_fallback(c, jd_profile, llm)
                results[c["candidate_id"]] = outcome
                if outcome["reasoning"].startswith("Automated scoring only"):
                    failed.append(c["candidate_id"])

        # Checkpoint after every batch — not just once at the end.
        _save_results(results, artifacts_dir)
        time.sleep(SLEEP_BETWEEN_BATCHES)

    print(f"\n\u2713 llm_evaluations.json saved ({len(results)} total candidates)")
    print(f"  Scored this run: {len(remaining) - len(failed)} | Failed: {len(failed)}")
    if failed:
        print(f"  Failed IDs: {failed}")
    return results


if __name__ == "__main__":
    run()
```

**What changed and why each change matters:**

| Change | Effect |
|---|---|
| `behavioral_signals` removed from LLM payload | Eliminates ~6 fields × every candidate × every batch. These are already scored deterministically in `behavioral.py` (see Step 7). The `EVAL_PROMPT` rubric never references them — this was pure token duplication |
| `notice_period_days` moved to `behavioral.py` (Step 7) | Preserves the signal without paying LLM tokens for it — scored for free alongside the other behavioral multipliers |
| `career_history`: full text only for 4 most recent roles | Single biggest per-candidate token saving. Older roles still contribute title/company/duration for pattern detection (title-chasing, tenure gaps) — just not full prose |
| `BATCH_SIZE` 5 → 8 | Fixed JD-context (role summary, requirements, disqualifiers, rubric) now paid ~37% as often for the same candidate volume |
| Model `llama-3.3-70b-versatile` → `llama-3.1-8b-instant` | Higher free-tier token quota on the step that runs at volume; 70B stays reserved for `analyze_jd.py` (runs once, quality matters most) |
| `estimate_max_tokens(batch_size)` | No longer flat 2000 regardless of actual batch size — sized to what the output actually needs |
| Retry-after parsing + `LONG_WAIT_THRESHOLD_S` | If Groq's error implies a daily cap, the script stops burning retries and tells you to resume later, rather than sleeping silently for minutes mid-batch |

> **Validate after switching to 8B:** the model swap is the change most likely to affect output quality, not just token count. After running Step 9, do the standard reasoning spot-check (10 random samples from `llm_evaluations.json`). If reasoning specificity degrades noticeably, tighten the prompt instructions further — not revert to 70B, which defeats the purpose.

### STEP 10 — Phase A: Final Score Assembly (`pre_computation/assemble_ranking.py`)

**Objective:** Combine the three score components into a final ranking. Store component scores separately so weights can be adjusted in Phase B without re-running the expensive LLM evaluation.

> **Inputs:** all three independently computed scores — `llm_score` (Step 9), `semantic_score` (Step 8), `behavioral_score` (Step 7/`behavioral.py`)  
> **Produces:** `final_score = 0.65 × llm_score + 0.20 × semantic_score + 0.15 × behavioral_score` (default weights)  
> **Also produces:** `rank_config.json` — records which weights generated this ranking. Edit this file and re-run `rank.py` to retune without repeating Phase A.  
> **Passes through unchanged:** `reasoning` string from `llm_evaluations.json` — `assemble_ranking.py` does not modify or regenerate it.

```python
# pre_computation/assemble_ranking.py
"""
Phase A - Step 5: Combine semantic, behavioral, and LLM scores into a final ranked list.

Public entrypoint: run(...) -> list[dict] (also saved to final_ranking.json)

Outputs:
  artifacts/final_ranking.json  — sorted array of top N candidates with all component scores
  artifacts/rank_config.json    — adjustable weights used by Phase B (rank.py)
"""
import json

ARTIFACTS_DIR = 'artifacts'
TOP_N = 300   # store top 300; Phase B selects top 100

# Starting weights — can be adjusted in rank_config.json without touching Python
DEFAULT_WEIGHTS = {
    "llm_weight":        0.65,   # primary — semantic reasoning by LLM
    "semantic_weight":   0.20,   # secondary — vector similarity
    "behavioral_weight": 0.15,   # tertiary — platform availability signals
}


def compute_final_score(llm_score: float, semantic_score: float, behavioral_score: float,
                         weights: dict) -> float:
    """Pure scoring function — trivial to unit test with hand-picked inputs."""
    return (
        weights['llm_weight']        * llm_score
        + weights['semantic_weight']  * semantic_score
        + weights['behavioral_weight'] * behavioral_score
    )


def assemble_records(evals: dict, behavioral_scores: dict, semantic_scores: dict,
                      weights: dict) -> list:
    """Combines the three score sources into one record per candidate. Pure — no file I/O."""
    records = []
    for cid, eval_data in evals.items():
        sem   = semantic_scores.get(cid, 0.0)
        beh   = behavioral_scores.get(cid, 0.5)
        llm   = eval_data['score']
        final = compute_final_score(llm, sem, beh, weights)
        records.append({
            "candidate_id":     cid,
            "llm_score":        round(llm,  6),
            "semantic_score":   round(sem,  6),
            "behavioral_score": round(beh,  6),
            "final_score":      round(final, 6),
            "reasoning":        eval_data['reasoning'],
        })
    # Sort by final_score DESC, candidate_id ASC for ties (V13 compliance)
    records.sort(key=lambda r: (-r['final_score'], r['candidate_id']))
    return records


def run(artifacts_dir: str = ARTIFACTS_DIR, top_n: int = TOP_N, weights: dict = None) -> list:
    weights = weights or DEFAULT_WEIGHTS

    evals = json.load(open(f'{artifacts_dir}/llm_evaluations.json'))
    behavioral_scores = json.load(open(f'{artifacts_dir}/behavioral_scores.json'))

    semantic_scores = {}
    with open(f'{artifacts_dir}/shortlist.jsonl') as f:
        for line in f:
            if line.strip():
                c = json.loads(line)
                semantic_scores[c['candidate_id']] = c['_semantic_score']

    records = assemble_records(evals, behavioral_scores, semantic_scores, weights)
    top_records = records[:top_n]

    with open(f'{artifacts_dir}/final_ranking.json', 'w') as f:
        json.dump(top_records, f, indent=2)
    with open(f'{artifacts_dir}/rank_config.json', 'w') as f:
        json.dump(weights, f, indent=2)

    print(f"✓ final_ranking.json: {len(top_records)} candidates")
    print(f"✓ rank_config.json: weights saved")
    print(f"\nTop 5 candidates:")
    for r in top_records[:5]:
        print(f"  {r['candidate_id']}  final={r['final_score']:.4f}  "
              f"llm={r['llm_score']:.3f}  sem={r['semantic_score']:.3f}  "
              f"beh={r['behavioral_score']:.3f}")
        print(f"  → {r['reasoning'][:120]}...")

    return top_records


if __name__ == '__main__':
    run()
```

### STEP 11 — Phase A Orchestrator (`pre_computation/pipeline.py`)

**Objective:** A single entry point that runs Steps 4–10's `run()` functions (`analyze_jd.run()` → `assemble_ranking.run()`) in order, by calling them directly as Python functions. This is the **only command you run** to execute all of Phase A — and reading this one file top to bottom tells you exactly what Phase A does, in what order, and what each stage produces, without needing to read the implementation of any individual step.

**Why direct function calls instead of `subprocess`:** The earlier design ran each step as a separate OS process via `subprocess.run([sys.executable, script])`. That works, but it actively gets in the way of debugging:

| | `subprocess.run(...)` | Direct function call (`step_fn(**kwargs)`) |
|---|---|---|
| On failure | You get an exit code and whatever the child process printed to stdout — no Python traceback in *this* process | The real exception propagates here, with a full traceback pointing at the exact line that failed |
| Debugging one step | Have to re-run the whole script from the command line, can't attach a debugger mid-flight | `breakpoint()` anywhere inside a step's `run()` stops execution right there, in this process |
| Testing a step in isolation | Have to build a full CLI invocation with argparse flags | `from pre_computation.build_shortlist import run; run()` from a REPL or notebook — done |
| Passing data between steps | Only via files on disk (which is still how artifacts are shared here — that's fine) plus parsed argv strings | Plain Python values — no serialization round-trip for anything that isn't already meant to be a file artifact |

The actual logic for "what does step 3 do" still lives entirely inside `build_shortlist.py` — `pipeline.py` itself contains almost no business logic, just a list of `(step_number, function, label, expected_outputs)` tuples and a loop that calls each function and checks whether its outputs already exist (so a partial failure doesn't force a full restart).

**Does the pipeline force re-run even if a step was already done?**

No. This is the `artifacts_exist()` check:

```python
def artifacts_exist(filenames: list) -> bool:
    return all((ARTIFACTS_DIR / f).exists() for f in filenames)
```

Each step declares which files it produces. If those files exist on disk, `run_step()` prints `⏭ Skipping` and moves on immediately — no re-computation. The embedding step (which takes 15–25 minutes) is only re-run if `artifacts/candidate_embeddings.npy` is deleted or `--force` is passed. This is what makes resume safe: if the LLM evaluation hits a token limit partway through, fix the issue and run `python -m pre_computation.pipeline --from 4` — Steps 1–3 are untouched.

```
Default run:      python -m pre_computation.pipeline
  → Steps 1-5 run only if their artifacts don't exist yet

Resume from 4:    python -m pre_computation.pipeline --from 4
  → Steps 1-3 skipped (artifacts exist), Steps 4-5 run

Force everything: python -m pre_computation.pipeline --force
  → All 5 steps re-run regardless of existing artifacts

Single step:      python -m pre_computation.pipeline --only 3
  → Only build_shortlist.run() is called (useful when tuning shortlist size)
```

```python
#!/usr/bin/env python3
# pre_computation/pipeline.py — PHASE A ORCHESTRATOR
"""
Runs the entire Phase A pre-computation pipeline end to end, in order, by
calling each step's run() function directly — NOT via subprocess.

  1. analyze_jd.run()       JD (.docx)               -> jd_profile.json, jd_embedding.npy
  2. embed_candidates.run() 100K candidates.jsonl  -> candidate_embeddings.npy, candidate_ids.txt
  3. build_shortlist.run()        embeddings + behavioral   -> shortlist.jsonl (top N), behavioral_scores.json
  4. evaluate_candidates.run()     shortlist + jd_profile    -> llm_evaluations.json (Groq LLM scores + reasoning)
  5. assemble_ranking.run()         all component scores      -> final_ranking.json, rank_config.json

This is the ONLY script you need to run for all of Phase A. Each underlying
step module is ALSO independently runnable on its own — via
`python -m pre_computation.analyze_jd`, `python -m pre_computation.build_shortlist`,
etc. — which is useful when iterating on just one step (see Step 15,
"Score Tuning"). This orchestrator is the recommended way to run all five
in sequence; it is not the only way to run any one of them.

Network: ON for steps 1 and 4 (Groq API calls). Steps 2, 3, 5 are local/CPU only.
Total runtime: ~45-60 minutes, dominated by step 2 (embedding) and step 4 (LLM eval).

Usage:
  python -m pre_computation.pipeline                                  # run everything
  python -m pre_computation.pipeline --jd data/job_description.docx   # explicit JD path
  python -m pre_computation.pipeline --force                          # re-run even if artifacts exist
  python -m pre_computation.pipeline --only 4                         # re-run a single step
  python -m pre_computation.pipeline --from 3                         # resume from a given step
"""
import argparse
import time
from pathlib import Path

from . import analyze_jd, embed_candidates, build_shortlist, \
              evaluate_candidates, assemble_ranking

ARTIFACTS_DIR = Path('artifacts')

# (step number, run() function, human label, output artifacts that mark this step done)
# The output list is what lets pipeline.py skip a step on resume instead of redoing it.
# Note: this is the ONLY place that lists what Phase A does — nothing else
# in this file knows or cares what's inside any individual step.
PIPELINE = [
    (1, analyze_jd.run,       'JD analysis (Groq LLM)',        ['jd_profile.json', 'jd_embedding.npy']),
    (2, embed_candidates.run, 'Embed all 100K candidates',     ['candidate_embeddings.npy', 'candidate_ids.txt']),
    (3, build_shortlist.run,        'Shortlist top N candidates',           ['shortlist.jsonl', 'behavioral_scores.json']),
    (4, evaluate_candidates.run,     'LLM evaluate shortlist (Groq)', ['llm_evaluations.json']),
    (5, assemble_ranking.run,         'Assemble final ranking',        ['final_ranking.json', 'rank_config.json']),
]


def artifacts_exist(filenames: list) -> bool:
    return all((ARTIFACTS_DIR / f).exists() for f in filenames)


def run_step(step_num: int, step_fn, label: str, kwargs: dict, force: bool, outputs: list) -> None:
    print(f"\n{'=' * 70}\nSTEP {step_num}/5 — {label}\n{'=' * 70}")

    if not force and artifacts_exist(outputs):
        print(f"  ⏭  Skipping — outputs already exist: {outputs}")
        print(f"     (use --force or delete the artifact to re-run)")
        return

    t0 = time.time()
    try:
        step_fn(**kwargs)
    except Exception:
        elapsed = time.time() - t0
        print(f"\n✗ STEP {step_num} FAILED after {elapsed:.0f}s — see traceback above")
        print(f"  Fix the error, then resume with: python -m pre_computation.pipeline --from {step_num}")
        raise   # re-raise — no subprocess boundary, so the real traceback is already visible

    print(f"  ✓ Step {step_num} done in {time.time() - t0:.0f}s")


def main():
    parser = argparse.ArgumentParser(description='Phase A orchestrator — runs steps 1-5 in order.')
    parser.add_argument('--jd', default='data/job_description.docx', help='Path to JD file (passed to step 1)')
    parser.add_argument('--force', action='store_true', help='Re-run every step even if artifacts already exist')
    parser.add_argument('--from', dest='from_step', type=int, default=1, help='Resume starting at this step (1-5)')
    parser.add_argument('--only', type=int, default=None, help='Run only this single step (1-5)')
    args = parser.parse_args()

    ARTIFACTS_DIR.mkdir(exist_ok=True)

    steps_to_run = PIPELINE
    if args.only:
        steps_to_run = [s for s in PIPELINE if s[0] == args.only]
    elif args.from_step > 1:
        steps_to_run = [s for s in PIPELINE if s[0] >= args.from_step]

    print(f"Phase A — running {len(steps_to_run)} step(s)")
    t_start = time.time()

    for step_num, step_fn, label, outputs in steps_to_run:
        kwargs = {'jd_path': args.jd} if step_num == 1 else {}

### STEP 12 — Phase B: Ranking Script (`rank.py`)

**Objective:** The Phase B entry point. Loads pre-computed artifacts, applies weights from `rank_config.json`, sorts by `(-final_score, candidate_id)`, and writes the submission CSV. Must run in under 5 minutes with no network access.

**This script is intentionally simple.** All intelligence lives in Phase A. Phase B is just loading a JSON and writing a CSV.

```python
#!/usr/bin/env python3
# rank.py — PHASE B ENTRY POINT
"""
Produces submission.csv from pre-computed artifacts.
No network access. No API calls. CPU only. Must finish in ≤5 minutes.

Usage:
  python rank.py --candidates ./data/candidates.jsonl --out ./submission.csv
  python rank.py --candidates ./data/candidates.jsonl --out ./submission.csv --artifacts ./artifacts
"""
import argparse, json, os, time
from util.submission import write_submission_csv

def main():
    t0 = time.time()

    parser = argparse.ArgumentParser()
    parser.add_argument('--candidates', required=True,
                        help='Path to candidates.jsonl — kept for reproduce-command compliance; '
                             'this script reads everything it needs from --artifacts instead.')
    parser.add_argument('--out',       required=True,  help='Output CSV path')
    parser.add_argument('--artifacts', default='artifacts', help='Artifacts directory')
    args = parser.parse_args()

    # ── Validate artifacts exist before attempting to load ─────────────────
    for fname in ['final_ranking.json', 'rank_config.json']:
        path = os.path.join(args.artifacts, fname)
        if not os.path.exists(path):
            raise SystemExit(
                f"\nMissing required artifact: {path}\n"
                f"Run Phase A first: python -m pre_computation.pipeline\n"
                f"Or resume from the relevant step: python -m pre_computation.pipeline --from 5"
            )

    # ── Load pre-computed ranking ───────────────────────────────────────────
    print(f"[{time.time()-t0:.1f}s] Loading pre-computed ranking...")
    with open(f'{args.artifacts}/final_ranking.json') as f:
        ranking = json.load(f)   # sorted array of dicts with component scores + reasoning

    with open(f'{args.artifacts}/rank_config.json') as f:
        weights = json.load(f)   # {llm_weight, semantic_weight, behavioral_weight}

    print(f"[{time.time()-t0:.1f}s] Loaded {len(ranking)} pre-ranked candidates")
    print(f"  Weights: LLM={weights['llm_weight']}, "
          f"Semantic={weights['semantic_weight']}, "
          f"Behavioral={weights['behavioral_weight']}")

    # ── Re-apply weights (allows tuning rank_config.json without re-running Phase A) ──
    for r in ranking:
        r['_recomputed_score'] = (
            weights['llm_weight']        * r['llm_score']       +
            weights['semantic_weight']   * r['semantic_score']  +
            weights['behavioral_weight'] * r['behavioral_score']
        )

    # ── Sort by (-recomputed_score, candidate_id) — enforces V13 tie-break ──
    ranking.sort(key=lambda r: (-r['_recomputed_score'], r['candidate_id']))

    # ── Build submission rows (top 100 only) ───────────────────────────────
    rows = []
    for rank_pos, r in enumerate(ranking[:100], start=1):
        rows.append({
            'candidate_id': r['candidate_id'],
            'rank':         rank_pos,             # native Python int — critical for V9
            'score':        round(r['_recomputed_score'], 6),
            'reasoning':    r['reasoning'],
        })

    # ── Write CSV (all format checks enforced inside write_submission_csv) ──
    write_submission_csv(rows, args.out)

    elapsed = time.time() - t0
    print(f"\n✓ Done in {elapsed:.1f}s → {args.out}")
    if elapsed > 270:
        print("⚠  WARNING: Approaching 5-minute limit!")

if __name__ == '__main__':
    main()
```

**Expected output:** A working `rank.py` that produces a valid CSV from artifacts in seconds.

### STEP 13 — CSV Formatter (`util/submission.py`)

**Objective:** Write the exact CSV format the validator expects. This module enforces all V1–V14 rules internally as assertions — a format failure fires before the file is written, not at upload time.

```python
# util/submission.py
import csv

def write_submission_csv(rows: list, output_path: str) -> None:
    """
    Writes submission CSV enforcing all validate_submission.py rules (V1–V14).

    V4:  Header exactly ['candidate_id', 'rank', 'score', 'reasoning']
    V6:  Exactly 100 data rows
    V7:  Exactly 4 columns per row (handled by DictWriter with exact fieldnames)
    V9:  rank must be a native Python int — csv.writer writes "1" not "1.0" or "01"
    V10: Ranks 1–100 each exactly once
    V11: score must be a float (ensured by round())
    V12: Scores non-increasing by rank
    V13: Equal scores → candidate_id ascending (tie-break rule)
    V3:  UTF-8 via open(..., encoding='utf-8')
    """
    assert len(rows) == 100, f"Expected 100 rows, got {len(rows)}"

    # Sort by rank before assertions (validator also sorts internally)
    by_rank = sorted(rows, key=lambda r: r['rank'])

    # V10: ranks must be 1–100 each exactly once
    ranks = [r['rank'] for r in by_rank]
    assert ranks == list(range(1, 101)), \
        f"Ranks must be exactly 1–100 in order. Got first/last: {ranks[0]}, {ranks[-1]}"

    # V9: rank must be a native Python int
    for r in by_rank:
        assert isinstance(r['rank'], int), (
            f"rank must be Python int, got {type(r['rank']).__name__}: {r['rank']!r}. "
            f"csv.writer would write '{str(r['rank'])}', which fails the V9 round-trip check."
        )

    # V12: scores must be non-increasing by rank
    scores = [r['score'] for r in by_rank]
    for i in range(1, len(scores)):
        assert scores[i] <= scores[i-1] + 1e-9, (
            f"Score not non-increasing at rank {i+1}: "
            f"score[rank {i}]={scores[i-1]} but score[rank {i+1}]={scores[i]}"
        )

    # V13: equal scores → candidate_id must be ascending
    for i in range(len(by_rank) - 1):
        r1, r2 = by_rank[i], by_rank[i+1]
        if abs(r1['score'] - r2['score']) < 1e-12:
            assert r1['candidate_id'] <= r2['candidate_id'], (
                f"V13 Tie-break violation: ranks {r1['rank']} and {r2['rank']} have equal scores "
                f"but {r1['candidate_id']} > {r2['candidate_id']}. "
                f"Fix: ensure rank.py sorts by (-score, candidate_id)."
            )

    # Write
    fieldnames = ['candidate_id', 'rank', 'score', 'reasoning']
    with open(output_path, 'w', newline='', encoding='utf-8') as f:
        writer = csv.DictWriter(f, fieldnames=fieldnames, quoting=csv.QUOTE_MINIMAL)
        writer.writeheader()
        writer.writerows(by_rank)

    print(f"✓ Submission written: {output_path} ({len(rows)} rows)")
    print(f"  Score range: [{scores[-1]:.6f}, {scores[0]:.6f}]")
```

### STEP 15 — Score Tuning & Quality Improvement

**Objective:** Improve ranking quality. The key insight is that tuning weights in `artifacts/rank_config.json` does not require re-running any Phase A scripts — only `rank.py` needs to be re-run. This makes weight iteration extremely fast.

**Tuning actions in priority order:**

1. **Weight adjustment** — edit `artifacts/rank_config.json` and re-run `rank.py` (takes seconds):
   ```json
   {
     "llm_weight": 0.65,
     "semantic_weight": 0.20,
     "behavioral_weight": 0.15
   }
   ```
   Try increasing `llm_weight` toward 0.70–0.75 if the LLM evaluations look accurate. Try increasing `behavioral_weight` if too many inactive candidates are appearing in the top 10. After each change, re-run `rank.py` and manually inspect the new top 10.

2. **LLM evaluation prompt refinement** — if the `artifacts/llm_evaluations.json` file contains generic or templated reasoning strings, the problem is in the `EVAL_PROMPT` / `build_eval_prompt()` in `evaluate_candidates.py`. Add a stronger instruction:
   ```
   IMPORTANT: Each reasoning string must be completely different. 
   Reference the specific company the candidate worked at and 
   the specific system or project mentioned in their description.
   Do not use phrases like "strong background in" or "extensive experience with".
   ```
   After changing the prompt, re-run only the two affected steps directly — no need to go through the full orchestrator:
   ```bash
   python -m pre_computation.evaluate_candidates
   python -m pre_computation.assemble_ranking
   ```
   (approximately 30 minutes total, depending on Groq's current rate limits). You do not need to re-run Steps 1–3.

3. **Shortlist coverage check** — verify that the LLM is not constrained by the shortlist. If the best candidates in the shortlist (by LLM score) are clustered at the bottom of the embedding similarity ranking, consider increasing the shortlist size:
   ```python
   import json
   evals = json.load(open('artifacts/llm_evaluations.json'))
   high_llm = {cid: e for cid, e in evals.items() if e['score'] >= 0.70}
   print(f"Candidates with LLM score ≥ 0.70: {len(high_llm)}")
   # If this is much less than 100, the shortlist may be too narrow
   # or the LLM is scoring conservatively — check a few manually
   ```
   If you do increase it, pass the new size directly to the function rather than editing the module constant:
   ```bash
   python -c "from pre_computation.build_shortlist import run; run(shortlist_size=3000)"
   python -m pre_computation.evaluate_candidates
   python -m pre_computation.assemble_ranking
   ```

4. **Reasoning quality audit** — sample 10 random reasoning strings from the final top 100 and read them. Ask:
   - Do any two sound identical in structure? If yes, refine the evaluation prompt.
   - Does reasoning for rank 90 sound appropriately concerned vs rank 1 which sounds strong? The LLM should naturally calibrate tone by score.
   - Does any reasoning mention a skill or company name not in the candidate's profile? This is hallucination — the prompt already includes "Only cite facts that appear in the candidate data provided"; if it's still happening, strengthen that instruction further.

**What does NOT require re-running Phase A from scratch:**
- Weight changes in `rank_config.json` → re-run only `rank.py`
- Changes to the evaluation prompt → `python -m pre_computation.evaluate_candidates` then `python -m pre_computation.assemble_ranking`
- Changes to shortlist size → `python -m pre_computation.build_shortlist` then steps 4 and 5

**What requires re-running from Step 2:**
- Changes to `build_candidate_text()` in `util/candidate_text.py` → `python -m pre_computation.embed_candidates` and all steps after

## Appendix — Validator Rules — Complete Reference

> `validate_submission.py` source has been reviewed in full. All 14 rules it enforces are listed here. Steps 13 and 16 implement these exactly.

| # | Rule | How it's enforced |
|---|---|---|
| V1 | Filename extension must be `.csv` (case-insensitive) | `path.suffix.lower() != ".csv"` |
| V2 | Filename stem must be non-empty (your registered participant ID) | `not path.stem` |
| V3 | File must be UTF-8 encoded | `open(..., encoding="utf-8")` → `UnicodeDecodeError` |
| V4 | Row 1 must be **exactly** `candidate_id,rank,score,reasoning` — no reorder, no extras | `header != REQUIRED_HEADER` (list equality) |
| V5 | Empty/blank rows are silently skipped | `if any(cell.strip() for cell in row)` |
| V6 | Exactly 100 non-empty data rows after the header | `n != 100` |
| V7 | Each data row must have exactly 4 columns | `len(cells) != 4` |
| V8 | `candidate_id`: non-empty, matches `^CAND_[0-9]{7}$`, no duplicates | regex + set check |
| V9 | `rank` must satisfy `str(int(rank_s)) == rank_s` — **`"01"`, `"1.0"`, `"1."` all fail** | explicit string round-trip |
| V10 | `rank`: integer 1–100, no duplicates, all 100 values must appear exactly once | range + set check |
| V11 | `score` must parse as `float()` | `float(score_s)` |
| V12 | Scores must be **non-increasing by rank** — validator sorts rows by rank first, then checks consecutive pairs | `s1 < s2 → error` |
| V13 | **Tie-break ⚠:** Equal scores at consecutive ranks → `candidate_id` must be ascending | `s1 == s2 and c1 > c2 → error` |
| V14 | Validator does **NOT** check that `candidate_id` values exist in `candidates.jsonl` — format only | (no such check in source) |

> **V13 implication:** Every sort that produces the final ranking must use `(-score, candidate_id)` as the sort key. IDs are zero-padded, so lexicographic order equals numeric order.

> **V9 implication:** Write `rank` as a Python native `int`. `csv.writer` calls `str()` on it, producing `"1"` which passes the round-trip check. Never pass a float or zero-padded string.
